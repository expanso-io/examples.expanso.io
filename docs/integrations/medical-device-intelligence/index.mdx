## Product boundary

- **Expanso:** reads configured inputs, invokes the command processor, shapes its response, and fans out output.
- **Custom boundary:** Python loads synthetic fixture files, calls Claude or emits mock JSON, and forwards candidates to an external endpoint. No medical-device protocol adapter is included.

## Pipeline

Inspect the canonical [`pipeline.yaml`](https://github.com/expanso-io/examples.expanso.io/blob/main/docs/integrations/medical-device-intelligence/pipeline.yaml). Its first mapping tags messages; the custom analyzer independently loads all three fixtures.

- `scripts/analyze_reports.py` loads fixtures, optionally calls Claude, and emits candidate review items without diagnostic conclusions.
- `scripts/mock_receiver.py` accepts local JSON; `scripts/test-pipeline.sh` validates fixtures, YAML, mock shape, and the receiver without executing Expanso or assessing model output.

## Limitations and assumptions

<Limitations>

- All records and outputs are synthetic. The mock groups evidence without diagnoses, severity, confidence, or recommended action; never use it for clinical, safety, maintenance, or purchasing decisions.
- With an API key, the custom analyzer sends the full fixture batch to Claude. Data minimization, retention, access control, residency, and vendor configuration are not assessed here.
- No current machine gate executes the full YAML, command, fan-out, retry, or failure path. This architecture establishes no legal, regulatory, security, or quality-system requirement.

</Limitations>
