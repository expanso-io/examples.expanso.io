# SCADA Edge Processing for Power Grids

:::caution Integration and compliance boundary
Expanso does not natively poll or decode Modbus, Profinet, OPC UA, DNP3, or IEC 61850 in this example. A protocol adapter or gateway must first emit the line-oriented records consumed by the pipeline. The field-removal and local-archive patterns can support a NERC CIP control program, but this tutorial does not determine which data is BES Cyber System Information or establish compliance.
:::

## The Blind Spot: Data Trapped Behind the OT Fence

Substations can produce frequent voltage, current, frequency, temperature, and power readings. The actual rate and the portion sent to analytics vary by equipment, polling policy, architecture, and operating requirements.

Why? Three compounding problems:

- **OT/IT air gaps** — SCADA networks are isolated for security. Getting data out requires crossing a carefully guarded boundary, which operations teams resist for good reason.
- **Volume and retention** — Shipping every raw reading can increase network and storage use; the right retention policy depends on the operational and investigative value of the data.
- **Security and governance** — Organizations may classify topology, relay, and network information as sensitive. Authorized owners must define what can cross each boundary and which NERC CIP requirements apply.

The result can be a gap between locally available telemetry and the subset approved for central analytics.

**The opportunity:** process approved telemetry near its source and send only the records required by each destination.

## Four Wins, One Pipeline

Expanso Edge can run on an industrial PC, edge gateway, or Kubernetes node in a substation DMZ. In this architecture, a separate protocol adapter decodes the OT protocol and emits records to the pipeline before the configured output crosses the OT/IT boundary.

### Win #1: Unlock Data Trapped in OT Networks

Modbus TCP, DNP3, and IEC 61850 are the languages of operational technology. SCADA historians understand them, but cloud analytics platforms don't. Traditional approaches require protocol converters, data historians, and bespoke middleware — each adding latency, cost, and another system to maintain.

The tutorial starts after protocol decoding and translates adapter output into structured JSON:

- **Protocol adapter required** — a gateway or adapter reads the RTU/PLC protocol and emits the tutorial's delimited records
  ✅ **TCP line input** — receives those adapter-emitted records; it is not a Modbus client or decoder
  ✅ **Register-to-field mapping** — REG 40001 becomes `voltage_kv`, with scaling applied automatically
  ✅ **Semicolon-delimited to structured JSON** — raw `REG=40001;VAL=14823` becomes `{ "voltage_kv": 148.23 }`
  ✅ **Explicit boundary** — protocol conversion remains a separate, testable integration responsibility

### Win #2: Edge ML for Fault Classification at Source

The example classifies readings with configurable static thresholds before routing. These illustrative thresholds are not protection settings and must not be used for control decisions without engineering validation.

Expanso Bloblang processors run fault classification logic directly at the substation:

✅ **VOLTAGE_DEVIATION** — transmission voltage outside ±10% of 132 kV nominal (110–145 kV band)  
✅ **FREQUENCY_DRIFT** — frequency outside the tutorial's illustrative 59.95–60.05 Hz band
✅ **THERMAL_OVERLOAD** — transformer or line temperature exceeding thermal rating (75°C threshold)  
✅ **LINE_FAULT** — current spike combined with voltage drop indicating fault current path

Classification happens before the output stage. Measure end-to-end latency and validate the classifications against approved site logic before relying on them operationally.

### Win #3: Policy-Based Data Minimization

The pipeline can remove fields selected by your security and compliance owners before a configured egress. Field names in this tutorial are examples; they are not a legal interpretation of NERC CIP or an exhaustive classification.

Expanso Edge handles this at the pipeline level:

✅ **Strip before egress** — topology fields (`bus_topology`, `relay_config`, `esp_network_map`) are deleted in the Bloblang processor before any data crosses the substation boundary  
✅ **Aggregate, don't export** — instead of raw register values, send KPI summaries. Cloud systems get `voltage_avg_kv`, `fault_count`, and `uptime_pct` — not the raw topology that generated them  
✅ **Local archive option** — retain an approved record locally when required by your evidence and retention policy
✅ **Configurable by policy** — the fields your authorized owners classify are configurable in pipeline YAML

### Win #4: Better Data Reaches the Historian

For data you _do_ send to the SCADA historian, Expanso makes it arrive cleaner, faster, and more useful:

✅ **Pre-parsed fields** — historian receives `voltage_kv: 148.23` instead of raw `REG=40001;VAL=14823`  
✅ **Normalized units** — all values scaled and typed consistently before arrival  
✅ **Deduplicated readings** — consecutive identical readings suppressed; historian sees state changes, not noise  
✅ **Enriched context** — `substation_id`, `region`, `device_id` added automatically from environment

## If You Know SCADA, You Already Know Expanso

Expanso uses familiar SCADA concepts — but as a programmable pipeline:

| **SCADA Concept**            | **Expanso Equivalent**                      | **What's Better**                                   |
| ---------------------------- | ------------------------------------------- | --------------------------------------------------- |
| RTU/PLC polling              | External protocol adapter                   | Adapter is required to decode the OT protocol       |
| Adapter output               | Pipeline `socket` input (line-oriented TCP) | Structured handoff into the transformation pipeline |
| SCADA historian              | Pipeline `output` (http_client)             | Multi-destination + pre-filtering                   |
| Manual alarm thresholds      | Bloblang `match` processors                 | Dynamic, programmable, version-controlled           |
| OT/IT data diode             | Expanso edge + field deletion               | Selective egress with audit trail                   |
| Proprietary historian format | Structured JSON                             | Open, queryable by any analytics tool               |

## What You'll Build

In this tutorial, you'll create a complete SCADA edge pipeline that:

1. **Receives adapter-decoded register records** via a line-oriented TCP socket
2. **Parses and normalizes** register addresses to engineering units (kV, A, Hz, °C, MW)
3. **Filters nominal readings** and classifies fault events at the edge
4. **Routes fault alerts** to the SCADA historian and PagerDuty; KPI summaries to Grafana cloud

## Prerequisites

- Expanso Edge installed ([quick start](https://docs.expanso.io/getting-started/quick-start/))
- A protocol adapter or the tutorial's synthetic line-record generator; a raw Modbus endpoint cannot connect directly to this pipeline
- Environment variables for historian URL, PagerDuty webhook, and Grafana Cloud
