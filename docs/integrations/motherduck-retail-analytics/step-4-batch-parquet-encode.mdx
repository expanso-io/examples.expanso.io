# Step 4: Batch and Encode to Parquet

Individual JSON transactions become compressed, columnar Parquet. Compression ratio and query performance depend on schema, values, row-group sizing, codecs, storage, and query shape; benchmark generated files before making performance claims.

## Batching Strategy

Choose Parquet object sizes from measured write, discovery, and query behavior. The reference uses an authored micro-batch:

```yaml
# Batch: collect 1000 transactions or flush every 10 seconds
- batch:
    count: 1000
    period: 10s
```

At 10 TPS per store, that's roughly one Parquet file every 100 seconds (1000 rows), or a flush every 10 seconds during slow periods. Both produce well-sized files for DuckLake.

## Flatten for Parquet

The example maps selected fields into flat columns and serializes nested items as JSON strings for later inspection:

```yaml
# Prepare for Parquet encoding
- mapping: |
    root = this
    # Serialize nested items array as JSON string for Parquet
    root.items_json = this.items.format_json()
    # Drop the original nested array
    root = root.without("items")
```

```yaml
- parquet_encode:
    schema:
      - name: txn_id
        type: UTF8
      - name: store_id

      - name: terminal_id
        type: INT32
      - name: timestamp
        type: UTF8

        type: UTF8
      - name: payment_method
        type: UTF8

        type: UTF8
      - name: items_json
        type: UTF8
      - name: item_count
        type: INT32
      - name: subtotal
        type: DOUBLE

```

## Measure Compression

Record raw JSON bytes, Parquet bytes, row count, schema, codec, row-group settings, and the exact generator seed for each run. The tutorial does not provide a verified compression ratio.

## Why Parquet + DuckLake?

- **Columnar**: Queries can select a field without reading unrelated columns, subject to the engine and file layout
- **Typed**: No runtime type parsing — DuckDB knows every column type at read time
- **Compressed**: Zstandard on columnar data compresses extremely well (repeated store IDs, payment methods, etc.)
- **DuckLake inlining**: Small batches get inlined into the catalog, periodically flushed to Parquet — no small-file overhead
