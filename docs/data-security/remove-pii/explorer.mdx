## Understanding the Stages

### Stage 1: Original Input
See the raw event with **all PII fields intact**. Notice the violations:
- `full_number` → PCI-DSS violation (credit card)
- `email`, `ip_address` → GDPR PII
- `user_name` → Direct identifier
- `latitude`, `longitude` → Precise location tracking

### Stage 2: Delete Payment Data
**First transformation:** Remove the highest-risk PII using `.without()`.
- Credit card numbers deleted (PCI-DSS compliance)
- Expiry dates removed
- `last_four` retained for fraud analytics

### Stage 3: Hash IP Address
**Second transformation:** Convert IP to SHA-256 hash.
- Original IP replaced with irreversible hash
- Same IP always produces same hash (uniqueness preserved)
- Enables abuse detection without storing actual IPs

### Stage 4: Hash Email
**Third transformation:** Hash email + extract domain.
- Email converted to hash for user counting
- Domain extracted separately (`example.com`)
- Allows organizational analytics (e.g., "20% users are @gmail.com")

### Stage 5: Pseudonymize User
**Fourth transformation:** Replace name with consistent user_id.
- `user_name` becomes `user_id` (e.g., "user_8f3e7a9c2d1b")
- Same user = same ID across all events
- Enables user journey tracking without knowing identity

### Stage 6: Generalize Location
**Final transformation:** Remove precise coordinates.
- GPS coordinates (`latitude`, `longitude`) deleted
- City and country preserved for regional analytics
- **Result:** Fully GDPR-compliant location data

## What You've Learned

After exploring all 6 stages, you now understand:

✅ **Deletion** - When to completely remove fields (`.without()`)
✅ **Hashing** - How to preserve uniqueness while removing reversibility
✅ **Pseudonymization** - Creating consistent IDs for tracking without identity
✅ **Generalization** - Reducing precision to balance privacy and analytics

## Common Questions

### Can I reverse the hash to get the original value?
No. SHA-256 is a one-way cryptographic function. Given a hash, you cannot compute the original value. This is **by design** - it's what makes the data GDPR-compliant.

### How do I count unique users if emails are hashed?
The hash function is **deterministic**: the same email always produces the same hash. Count distinct `email_hash` values to get unique user counts.

### What if I need to contact a user later?
Store the mapping (email → hash) in a separate, highly-secured system with strict access controls. Never store this mapping with your analytics data.

### Do I need all 5 transformation steps?
No. Choose based on your compliance requirements:
- **PCI-DSS only?** → Use Step 1 (delete credit cards)
- **GDPR minimal?** → Use Steps 1-3 (delete cards, hash IPs/emails)
- **Full compliance?** → Use all 5 steps
