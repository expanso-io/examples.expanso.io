# Cross-Border GDPR Compliance

Apply example data-minimization and pseudonymization controls at the edge before sending selected fields to a global analytics system.

:::caution Legal scope
This tutorial is a technical pattern, not a determination that its output is anonymous or that a transfer complies with GDPR. Salted hashes and generalized values can remain personal data when a person is still identifiable or the controller retains additional information. Have privacy and legal owners assess identifiability, lawful basis, transfer mechanisms, retention, access, and the complete processing context.
:::

## The Problem

Your organization has:

- EU customer transaction data subject to GDPR
- Global analytics platform (BigQuery in US or multi-region)
- Requirement to aggregate data globally for business intelligence

**The challenge:** GDPR Chapter V governs transfers of personal data to third countries or international organizations. The applicable transfer mechanism and safeguards depend on the actual parties, destination, and processing context.

**The pattern:** Delete fields that are not needed, pseudonymize selected identifiers, generalize high-precision values, and block records that fail the configured field checks. These controls can reduce exposure; they do not by themselves make data anonymous or authorize a transfer.

## The Pattern: 6 Data-Protection Steps

This pipeline demonstrates transformations that must be reviewed against a specific dataset and threat model:

### 1. **Tag Data Origin** вҶ’ Compliance Tracking

- Record source region, extraction time, pipeline ID
- Required for audit trail and data lineage

### 2. **Create GDPR Record** вҶ’ Legal Documentation

- Document lawful basis before transformation
- List all PII fields being processed
- Cite relevant GDPR article (44)

### 3. **Delete High-Risk Fields** вҶ’ No Recovery Needed

- Full names, addressesвҖ”no analytics value
- Convert DOB to age buckets (25-34, 35-44)

### 4. **Hash Identifiers** вҶ’ Pseudonymization

- Customer ID вҶ’ salted hash (for cohort counting)
- Email вҶ’ domain only (B2B vs B2C analysis)
- IBAN вҶ’ country code (geographic distribution)
- IP вҶ’ /16 subnet (regional analysis)

### 5. **Generalize Values** вҶ’ Reduce Precision

- Transaction amounts вҶ’ buckets
- Timestamps вҶ’ hour-level

### 6. **Validate Configured Fields** вҶ’ Technical Gate

- Verify no PII fields remain
- Record which configured checks ran

## What This DemonstratesвҖ”and Does Not Prove

The example removes direct fields and changes the precision or representation of others. The remaining output can still be personal data: stable hashes enable linkage, generalized values can combine with other datasets, and the controller may retain the source data or salt.

- **Direct-field handling** - Some fields are deleted and others are pseudonymized with salted hashes
- **Precision reduction** - Age and amount values are generalized into configured ranges
- **Linkage remains possible** - Stable pseudonyms and combinations of fields require an identifiability assessment
- **Cryptographic scope** - Salted hashing is one-way in this example, but dictionary attacks, salt access, and source-data linkage remain part of the threat model

**Result:** The output has a smaller set of direct identifiers and a machine-readable record of the configured transformations. Whether it is anonymous, and whether a cross-border transfer is lawful, requires a separate documented assessment.
